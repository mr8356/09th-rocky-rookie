# 1. Bash 셸의 기본 구조 이해

> Bash는 CLI에서 사용자가 입력한 명령을 읽고 해석해서 실행하는 쉘이고 , 단순히 하나의 명렁을 치는 도구가 아니라 명령들을 묶어서 반복 실행하고 자동화할 수 있기 때문에 중요한 것
> 

### Bash란 무엇인가

Bash는 Unix 계열 운영체제에서 사용되는 대표적인 셸(shell)이다.

이름은 **Bourne Again SHell**의 약자이며, 기존의 Bourne Shell(sh)을 기반으로 확장된 셸이다. Linux와 macOS 같은 환경에서 널리 사용되며, 많은 시스템에서 기본 셸 또는 표준적인 셸로 취급된다.

셸은 사용자가 입력한 명령어를 해석하고 실행해서, 사용자가 운영체제와 상호작용할 수 있도록 해 주는 프로그램이다.

즉, 사용자는 셸에 명령을 입력하고, 셸은 그 명령을 해석한 뒤 운영체제에 전달하여 실행 결과를 다시 사용자에게 보여준다.

Bash는 단순히 명령어를 한 줄씩 입력하는 도구가 아니라, 여러 명령을 조합하고 반복 실행하며 자동화할 수 있는 환경이라는 점에서 중요하다.

### Bash가 동작하는 기본 구조

Bash의 기본 구조는 다음과 같이 볼 수 있다.

```
사용자 → Bash 셸 → 커널 → 하드웨어
```

사용자가 터미널에 명령어를 입력하면, Bash 셸이 그 명령을 읽고 해석한다.

그 다음 실제 실행이 필요한 작업은 운영체제의 핵심인 커널에 요청한다.

커널은 파일 시스템, 프로세스, 메모리, 장치 같은 시스템 자원을 관리하면서 해당 작업을 수행하고, 그 결과를 다시 Bash를 통해 사용자에게 전달한다.

즉, Bash는 **사용자와 운영체제 사이의 중간 다리 역할**을 한다고 이해하면 된다.

### Bash의 명령 처리 흐름

Bash에서 명령이 처리되는 흐름은 보통 다음과 같다.

```
1. 명령어 입력
2. 명령어 해석
3. 실행
4. 결과 출력
```

조금 더 풀어서 보면 다음과 같다.

**1. 명령어 입력**

사용자가 터미널에서 `ls`, `cd`, `pwd` 같은 명령어를 입력한다.

**2. 명령어 해석**

Bash는 입력된 문자열을 읽고, 이것이 어떤 명령인지 판단한다.

이 과정에서 명령어와 옵션, 인자(argument)를 구분하고, 변수 치환이나 와일드카드 확장 같은 작업도 수행한다.

```
ls -l *.txt
```

Bash가 `ls`는 명령어, `-l`은 옵션, `*.txt`는 조건에 맞는 파일 목록으로 확장해서 해석한다.

**3. 실행**

해석이 끝나면 Bash는 명령을 실행한다.

이때 실행 대상은 크게 두 가지다.

- **내부 명령(builtin command)**: Bash 자체가 직접 수행하는 명령
예: `cd`, `echo`, `export`
- **외부 명령(external command)**: 시스템에 있는 실행 파일을 찾아 실행하는 명령
예: `ls`, `cp`, `cat`

예를 들어 `cd`는 현재 셸 자신의 작업 디렉토리를 바꿔야 하므로 Bash 내부 명령으로 동작하고, `ls`는 `/bin/ls` 같은 외부 프로그램을 실행하는 방식으로 동작한다.

**4. 결과 출력**

명령 실행 결과는 표준 출력이나 표준 에러 형태로 터미널에 표시된다.

사용자는 이 결과를 보고 다음 작업을 이어간다.

### Bash가 제공하는 핵심 기능

Bash의 기본 구조를 이해했다면, 그 위에서 어떤 기능을 제공하는지도 함께 알아두는 것이 좋다.

1. **명령어 실행**
    
    가장 기본적인 기능이다. 사용자는 Bash에서 시스템 명령어를 실행하고 프로그램을 시작할 수 있다.
    
2. **파일 시스템 탐색**
    
    디렉토리 이동, 파일 확인, 생성, 삭제 같은 작업을 수행할 수 있다.
    
    예: `cd`, `ls`, `mkdir`, `rm`
    
3. **환경 변수 관리**
    
    셸 세션이나 프로그램 실행에 필요한 값을 변수 형태로 저장하고 활용할 수 있다.
    
    예: `PATH`, `HOME`, `USER`, `SHELL`
    
4. **파이프와 리다이렉션**
    
    한 명령의 출력을 다른 명령의 입력으로 넘기거나, 출력을 파일에 저장하는 등의 데이터 흐름 제어가 가능하다.
    
    예: `|`, `>`, `>>`, `<`
    
5. **스크립팅과 자동화**
    
    여러 명령을 하나의 파일에 작성해 반복 작업을 자동화할 수 있다.
    
    이 점 때문에 Bash는 시스템 관리와 운영 자동화에서 매우 중요하다.
    

### 셸과 커널의 관계

Bash를 이해할 때 자주 같이 나오는 개념이 커널이다.

커널은 운영체제의 핵심 부분으로, 프로세스 실행, 메모리 관리, 파일 시스템 관리, 장치 제어 같은 일을 담당한다.

사용자는 커널에 직접 명령을 내리지 않고, 보통 Bash 같은 셸을 통해 간접적으로 작업을 요청한다.

즉, 관계를 정리하면 다음과 같다.

- **사용자**: 명령을 입력하는 주체
- **셸(Bash)**: 명령을 해석하고 실행 요청을 전달
- **커널**: 실제 시스템 자원을 관리하며 작업 수행

이 구조를 이해하면, 왜 Bash가 운영체제 사용의 출발점처럼 여겨지는지 자연스럽게 보인다.

### Bash 실행 형태

Bash는 상황에 따라 서로 다른 방식으로 실행될 수 있다.

이 부분은 Bash의 “구조”라기보다는 “동작 환경”에 가깝지만, 기본 이해를 위해 같이 알아두면 좋다.

1. **대화형 셸**
    
    사용자가 터미널에서 직접 명령을 입력하는 형태다.
    
    프롬프트(`$`, `#`)가 보이고, 입력 즉시 결과를 확인할 수 있다.
    
2. **비대화형 셸**
    
    스크립트 파일처럼 미리 작성된 명령을 자동으로 실행하는 형태다.
    
    사람이 한 줄씩 입력하지 않아도 동작한다.
    
3. **로그인 셸과 비로그인 셸**
    
    Bash는 실행 방식에 따라 읽는 설정 파일이 다를 수 있다.
    
    - 로그인 셸: `/etc/profile`, `~/.bash_profile` 등을 읽음
    - 비로그인 셸: `~/.bashrc` 등을 읽음
    
    이 차이는 사용 환경 초기화나 환경 변수 설정에서 중요하다.
    

### Bash와 관련된 기본 정보

다음 항목들은 Bash를 사용할 때 자주 확인하는 정보들이다.

| 항목 | 의미 |
| --- | --- |
| `/bin/bash` | Bash 실행 파일 위치 |
| `/etc/shells` | 시스템에서 사용 가능한 셸 목록 |
| `$SHELL` | 사용자의 기본 셸 경로 |
| `chsh` | 기본 셸 변경 명령 |

단, `$SHELL`은 “현재 실행 중인 셸”이라기보다 **기본 로그인 셸 정보**를 보여주는 경우가 많아서, 현재 실제 셸을 정확히 확인할 때는 `ps`나 `echo $0` 같은 방법을 같이 보기도 한다.

이 부분은 한 단계 더 깊게 들어간 내용이라 같이 알아두면 좋다.

### Bash가 중요한 이유

Bash가 중요한 이유는 단순히 명령어를 입력하는 창이기 때문이 아니라,

운영체제 사용의 기본 통로이면서 동시에 자동화 도구이기 때문이다.

사용자는 Bash를 통해 시스템을 직접 제어할 수 있고, 반복되는 작업을 스크립트로 작성해 자동으로 처리할 수 있다.

그래서 Bash는 서버 관리, 개발 환경 구성, 로그 처리, 배치 작업, 시스템 점검 같은 다양한 분야에서 널리 활용된다.

즉, Bash는 단순한 입력 도구가 아니라 **운영체제를 다루는 기본 인터페이스이자 자동화의 출발점**이라고 볼 수 있다.

# 2. 변수와 환경 변수 활용

### 변수란 무엇인가

변수는 값을 저장해두고 필요할 때 꺼내 쓰기 위한 이름이다.

Bash에서는 문자열, 숫자, 경로 같은 다양한 값을 변수에 저장할 수 있음 

```
name="yunseo"
```

이렇게 하면 `name`이라는 변수에 `"yunseo"`라는 값이 저장됨

```
echo $name
```

 `$`는 “이 변수의 값을 꺼내라”는 의미

### 변수의 기본 특징

- 변수 선언 시 공백이 없어야 한다
(`name = yunseo` X / `name=yunseo` O)
- 기본적으로 문자열로 취급된다
- 사용할 때는 `$`를 붙여야 한다

### 변수는 왜 쓰는가

변수는 반복되는 값을 저장하고 재사용하기 위해 사용

경로를 여러 번 사용할 때:

```
dir="/home/user/project"
cd $dir
ls $dir
```

이렇게 하면 값을 한 번만 정의하고 계속 사용 가능함 

### 환경 변수란 무엇인가

환경 변수는 **현재 셸뿐만 아니라, 실행되는 다른 프로그램에서도 사용할 수 있는 변수이다** 

- 일반 변수 → 현재 Bash 안에서만 사용
- 환경 변수 → 다른 프로그램까지 전달됨

### 환경 변수의 대표 예시

Bash에는 이미 정의된 환경 변수들이 있다.

```
echo $HOME
echo $PATH
echo $USER
echo $SHELL
```

- `$HOME` : 사용자 홈 디렉토리
- `$PATH` : 명령어를 찾는 경로 목록
- `$USER` : 현재 사용자 이름
- `$SHELL` : 기본 셸 경로

### PATH

**PATH 덕분에 우리는 전체 경로 안 쓰고 명령어 이름만 써도 되는 것** 

우리가 `ls`라고 입력하면 Bash는 다음처럼 동작한다 

- `$PATH`에 정의된 경로들을 순서대로 뒤져서
- `ls`라는 실행 파일을 찾고 실행한다

```
ls → /bin/ls 찾음 → 실행
```

 →  `$PATH`가 없으면 명령어를 못 찾는다

### 환경 변수 설정 방법

환경 변수는 `export`를 사용해서 설정한다.

```
export NAME="yunseo"
```

→ 현재 셸 + 그 셸에서 실행되는 프로그램에서도 사용 가능

### 일반 변수 vs 환경 변수 차이

| 구분 | 일반 변수 | 환경 변수 |
| --- | --- | --- |
| 범위 | 현재 셸 내부 | 자식 프로세스까지 전달 |
| 선언 | `name=value` | `export name=value` |
| 사용 | `$name` | `$name` |
- export를 붙이면 밖에서도 쓸 수 있는 변수가 됨 !

### 변수 확장

**변수 확장 = `$변수`를 실제 값으로 바꿔서 실행하는 것**

먼저 **해석해서 바꾼 다음** 실행함

```python
name="yunseo"
echo $name
-> 결과 : yunseo 
```

```python
echo $HOME
-> echo /Users/username 와 동일
```

```
file="test.txt"
rm $file
```

### 변수 사용 시 주의점

**1. 공백 문제**

```
name= yunseo   X 
name=yunseo     O 
```

**2. 따옴표**

```
name="yun seo"
```

- 공백 포함 문자열은 따옴표 필요

### 변수에서 시작해서 환경 변수가 확장되는 흐름

먼저 Bash에서 가장 작은 범위는 **현재 셸 안에서만 쓰는 변수**다.

```
name="yunseo"
```

이건 그냥 지금 내가 쓰고 있는 Bash만 아는 값이다. 스크립트 안에서 계산하거나, 파일 이름이나 경로를 잠깐 저장해 두는 용도로 쓰는 값이라고 생각하면 된다.

```
dir="/home/user/project"
cd$dir
ls$dir
```

이건 전부 **현재 Bash 안에서만** 의미가 있다. 첫 단계는 현재 Bash 안에서만 쓰는 값

**`export로 범위 확장`**

여기서 `export`를 붙이면 의미가 달라진다.

```
exportHTTP_PROXY=http://192.168.0.1:3128
```

이건 단순히 “값을 저장했다”가 아니라,

**내가 지금부터 실행할 다른 프로그램도 이 값을 받을 수 있게 하겠다**는 뜻이다.

즉 범위가 이렇게 커진다.

```
현재 Bash
→ Bash가 실행한 다른 프로그램까지
```

이게 환경 변수의 핵심이다.

Red Hat 문서의 Podman 예시도 정확히 이 구조다. Podman은 `HTTP_PROXY` 환경 변수를 읽어 프록시 정보를 확인하고, 이 값은 환경 변수로 직접 주거나 `/etc/profile.d` 아래에서 설정할 수 있다고 설명한다.

### `/etc/profile.d/*.sh`

- 그런데 매번 export 하기 귀찮으면? 여기서 `/etc/profile.d/*.sh`가 등장한다.

예를 들어 Red Hat 문서에는 이런 형태가 나온다.

```
# /etc/profile.d/unauthenticated_http_proxy.sh
exportHTTP_PROXY=http://192.168.0.1:3128
exportHTTPS_PROXY=http://192.168.0.1:3128
exportNO_PROXY=example.com,172.5.0.0/16
```

이걸 넣어두면 사용자가 로그인하거나 셸이 초기화될 때 이 파일이 읽히고, 그 안의 `export`가 자동으로 실행된다. 그래서 사용자가 터미널에서 시작한 Podman 같은 프로그램은 그 값을 상속받을 수 있다.

즉 이건 사실 새로운 개념이라기보다, **“export를 로그인 셸이 시작될 때 자동으로 해주는 장치”이다.** 

```
1. 변수
   → 현재 Bash 안에서만 사용

2. export
   → Bash가 실행하는 프로그램에도 전달

3. /etc/profile.d/*.sh
   → 로그인 시 export를 자동 적용
```

여기까지는 전부 **셸 중심** 이야기다.

### systemd의 EnvironmentFile

***But ! 문제는 모든 프로그램이 사용자의 셸에서 시작되는 게 아니라는 점이다.***

예를 들면: 

- 부팅할 때 자동 시작되는 서비스
- 백그라운드 데몬
- `systemctl start`로 띄운 서비스

이런 것들은 **사용자 로그인 셸을 안 거칠 수 있다**.

그래서 `/etc/profile.d`에 넣었다고 해서 무조건 다 적용되는 건 아니다.

이 방식은 어디까지나 **셸 초기화 계층**이기 때문이다. Podman 프록시 문서의 `/etc/profile.d` 예시는 “셸 기반 설정”에 해당한다.

**그래서 systemd의 EnvironmentFile이 필요하다**

Red Hat 이미지 모드 문서에서는 프록시 환경을 `/etc/example-proxy.env` 같은 파일에 모아 두고, 서비스 쪽에서 이 파일을 직접 읽게 한다.

예시는 이런 식이다.

```
[Service]
EnvironmentFile=/etc/example-proxy.env
```

문서에는 `bootc-fetch-apply-updates.service`용 드롭인에서 이런 구성을 보여주고, 같은 방식으로 Podman systemd 단위에도 `EnvironmentFile=/etc/example-proxy.env`를 추가할 수 있다고 설명한다.

이제는 셸이 export 해주는 게 아니라,

**systemd가 서비스를 실행할 때 직접 환경 파일을 읽어서 그 서비스 프로세스에 넣어주는 것**이다.

즉 마지막 단계는 이렇게 된다.

```
4. systemd의 EnvironmentFile=
   → 셸을 안 거치는 서비스에도 직접 적용
```

### 전체 흐름

```
현재 Bash에서만 쓸 값
→ export로 Bash가 실행한 프로그램까지 전달
→ /etc/profile.d로 로그인 셸 전체에 자동 적용
→ systemd EnvironmentFile로 백그라운드 서비스까지 적용
```

 `/etc/profile.d`와 `EnvironmentFile=`은 **적용 대상이 다른 두 방식이다** 

- `/etc/profile.d`는 사용자 셸에서 시작한 프로그램용
- `EnvironmentFile=`은 systemd 서비스용

## Podman 문서의 USER, GROUP 예제 정리

> 스크립트는 “로직”만 가지고 있다, 실제 값은 “밖에서 넣어준다” 그리고 그 값을 이용해서:조건 분기 , 권한 검사 ,실행 여부 판단 같은 작업을 수행한다.
**스크립트를 재사용할 수 있다**
> 
- 지금까지의 방식 내가 값을 만들고, 내가 전달함.
    - 내가 Bash에서 변수 만든다
    - `export` 해서 다른 프로그램에 넘긴다

Podman은 스크립트를 실행하기 전에 `USER`, `GROUP` 같은 값을 환경 변수로 미리 설정한다. 스크립트는 이 값을 직접 만들지 않고, 이미 주어진 값을 읽어서 사용한다.

```
id -nG "$USER" |grep -qw "$GROUP"
```

이 코드는 특정 사용자가 특정 그룹에 속해 있는지를 확인하는 로직이다.

`$USER`는 검사 대상 사용자이고, `$GROUP`은 포함 여부를 확인할 그룹이다.

중요한 점은 이 값들이 스크립트 내부에서 정의된 것이 아니라, Podman이 실행 전에 외부에서 주입한 환경 변수라는 점이다. 

즉, 실행 흐름은 다음과 같다.

- Podman이 스크립트를 실행하기 전에 필요한 값을 환경 변수로 설정한다
- 스크립트는 `$USER`, `$GROUP` 값을 읽는다
- 해당 값을 기준으로 조건을 검사한다
- 결과에 따라 성공 또는 실패를 반환한다

이 구조를 통해 스크립트는 특정 값에 종속되지 않고, 실행 환경에 따라 동작을 바꿀 수 있다.

환경 변수는 단순히 값을 저장하는 것이 아니라, 실행 환경이 스크립트에 전달하는 입력값 역할을 할 수 있다. 스크립트는 이 값을 활용해 조건 분기, 권한 검사, 실행 제어 등의 로직을 수행한다.

따라서 환경 변수는 “설정값”이면서 동시에 “외부에서 주어지는 실행 정보”라는 성격을 가진다.

### bootc / image mode

> 
> 
> 
> bootc / image mode에서는 이미지 안의 ENV보다, **systemd 설정이 더 우선이고 중요하다**
> 

이미지 모드 문서에는 `rhel-10-bootc` 이미지가 시스템에 설치된 경우 OCI의 `ENV`는 무시되고, 글로벌 시스템 환경은 `systemd` 구성을 바꿔 구성한다고 설명한다. 즉, 컨테이너 이미지 수준의 환경 설정과 실제 시스템 서비스 수준의 환경 설정이 완전히 동일하지 않다

초보자 입장에서는 “환경 변수는 어디서든 똑같이 먹는 값”처럼 느껴질 수 있는데, 실제로는 **실행 맥락**에 따라 적용 방식이 달라진다.

- 셸 스크립트에서의 환경 변수
- 컨테이너 런타임이 주입하는 환경 변수
- systemd 서비스가 읽는 환경 파일
- 이미지 메타데이터의 `ENV`

이건 비슷해 보여도 전부 동일한 계층은 아니다. Red Hat 문서는 특히 시스템 운영 관점에서 `systemd` 기반 관리가 중요하다는 점을 강조한다.

# 3. 표준 입력·출력·리다이렉션과 파이프 처리

셸에서 명령어 하나를 실행하면, 그 명령은 보통 **세 개의 통로**를 가진다 

- **stdin**: 입력이 들어오는 통로
- **stdout**: 정상 결과가 나가는 통로
- **stderr**: 오류 메시지가 나가는 통로
    - 사용자가 키보드로 값을 입력한다 → **stdin**
    - 프로그램이 계산 결과를 화면에 보여준다 → **stdout**
    - 파일이 없거나 명령이 잘못되면 에러를 띄운다 → **stderr**

셸에서는 이 셋을 각각 **표준 입력, 표준 출력, 표준 오류**라고 부른다. RHEL 문서에서도 이 세 가지를 파일 디스크립터 흐름으로 다루고 있고, 여러 예시에서 stdout과 stderr를 따로 처리하는 방식이 반복해서 나온다.

**프로그램은 원래 정해진 통로로 입출력을 하는데, 셸은 그 통로를 다른 곳으로 돌려보낼 수 있다.**

**→ 이게 `리다이랙션` !** 

**한 명령의 출력을 다음 명령의 입력으로 연결하는게 `파이프라인` !** 

### 리다이렉션

1. **`>` : stdout을 파일로 보내기**

 `> :` **표준 출력(stdout)** 의 도착지를 **파일**로 바꾸는 기호 

```
nmcli --offline connection add ... > /etc/NetworkManager/system-connections/example.nmconnection
```

 `nmcli --offline ...`가 만든 결과를 화면에 보여주는 대신 파일에 저장한 것

문서도 `nmcli`가 오프라인 모드에서 **표준 출력을 통해 키 파일 연결 프로필을 생성한다**고 설명하고, 그 결과를 `.nmconnection` 파일로 리다이렉션하는 예시를 직접 보여준다.

1. **`2>` : stderr를 따로 보내기**

`2>:`**표준 오류(stderr)** 의 방향 바꾸기 

```
# 이 사용자가 특정 그룹에 속해 있는지 검사하는 코드 
if id -nG "$USER" 2> /dev/null | grep -qw "$GROUP" 2> /dev/null ; then
    exit 0
else
    exit 1
fi
```

- `>` 는 보통 stdout
- `2>` 는 stderr

이렇게 사용하는 이유 :  `id`나 `grep`이 에러를 내더라도 그 에러 문구를 화면에 보이지 않게 따로 처리할 수 있음

1.  **`/dev/null` : 버리는 곳**

`/dev/null`은 에러 메시지를 보여주지 말고 그냥 버리라는 의미 

```
2> /dev/null
```

- `>` : 덮어쓰기
- `>>`: 추가 쓰기
    
    `echo hello >> log.txt`
    
- `<` : 입력 리다이랙션 
`command < input.txt`

### tee

보통 `>`를 쓰면 파일에는 저장되지만, 화면에서는 바로 못 보기때문에 화면에도 보고 파일에도 저장하고 싶을때 `tee`를 사용

```
strace ... |& tee your_log_file.log
```

### 파이프

`|`: 한 명령의 출력을 다음 명령의 입력으로 넘기기

```
A | B
```

**A의 출력(stdout)을 B의 입력(stdin)으로 넘기겠다는 뜻** 

- **예시 1 :** 아까 명령어 예시
    
    ```
    id-nG"$USER" |grep-qw"$GROUP"
    ```
    
    1. `id -nG "$USER"`
    → 이 사용자가 속한 그룹 이름들을 출력함
    2. `|`
    → 그 출력을 다음 명령으로 넘김
    3. `grep -qw "$GROUP"`
    → 그 목록 안에 원하는 그룹이 있는지 검사함
- **예시 2** :  필터링을 여러 단계로 이어 붙이기
    
    ```
    ... | grep " uid:" | cut -d ":" -f 2 | while read uid; do ...; done
    ```
    
    1. 앞 명령이 긴 결과를 뱉는다
    2. `grep " uid:"`
    → 필요한 줄만 고른다
    3. `cut -d ":" -f 2`
    → `:` 기준으로 잘라서 두 번째 칸만 뽑는다
    4. `while read uid; do ...; done`
    → 뽑아낸 값을 한 줄씩 읽으면서 반복 작업을 한다
    
    **“출력 → 필요한 줄만 선택 → 필요한 부분만 추출 → 한 줄씩 반복 처리”**
    

셸 스크립트가 단순히 명령 한 줄 치는 용도가 아니라 **출력을 가공해서 다음 작업으로 연결하는 자동화 도구**라는 점을 잘 보여주기 때문에 파이프라인이 중요하다 

### `|&`

- `|` 는 stdout만 다음 명령으로 넘김
- `|&` 는 **stdout + stderr 둘 다** 다음 명령으로 넘김

화면에서 다 보면서 로그 파일에도 남기기 위해서 ! 

# 4. 조건문과 반복문을 활용한 기초 스크립트 작성

## 조건문

**`if`**

```
if 조건
then
    실행할 작업
else
    다른 작업
fi
```

Bash의 `if`는 **명령 실행 결과가 성공했는지 실패했는지**를 기준으로 판단한다 

```
if id -nG "$USER" | grep -qw "$GROUP"
then
    echo "허용"
else
    echo "거부"
fi
```

1. `id -nG "$USER"`로 사용자의 그룹 목록을 출력하고
2. `grep -qw "$GROUP"`로 그 그룹이 있는지 확인한 뒤
3. 있으면 `then`, 없으면 `else`로 간다

사용 예시 

- 사용자가 특정 그룹에 속해 있으면 `exit 0`, 아니면 `exit 1`을 반환해 **실행 허용 여부 판단**
- 명령 실행 결과를 바탕으로 권한 검사, 파일 존재 검사, 서비스 상태 검사 같은 운영 작업 제어

**`case`**

> “지금 무슨 이벤트가 들어왔는지 보고, 이벤트 종류마다 다른 작업을 하자.”
> 

`if`가 “참이냐 거짓이냐” 중심이라면, `case`는 **값이 무엇이냐에 따라 여러 갈래로 분기**할 때 사용한다 

```
case 값 in
    패턴1)
        작업1
        ;;
    패턴2)
        작업2
        ;;
    *)
        기본 작업
        ;;
esac
```

예시 

```
case ${2:-NOTANEVENT} in
    block_all)
        ...
        ;;
    AP-STA-CONNECTED | CTRL-EVENT-EAP-SUCCESS | CTRL-EVENT-EAP-SUCCESS2)
        ...
        ;;
    AP-STA-DISCONNECTED | CTRL-EVENT-EAP-FAILURE)
        ...
        ;;
    allow_all)
        ...
        ;;
    NOTANEVENT)
        ...
        ;;
esac
```

 `hostapd_cli`가 전달한 이벤트 값을 보고

- `block_all`이면 모든 브리지 트래픽을 막고
- 인증 성공 관련 이벤트면 MAC 주소를 허용 목록에 넣고
- 인증 실패나 연결 해제 이벤트면 허용 목록에서 제거하고
- `allow_all`이면 다시 전체 허용하고
- 이벤트가 없으면 잘못 호출되었다는 메시지를 출력한다.

네트워크 이벤트 처리, 명령행 옵션 처리, 서비스 상태 분기, 문자열 값에 따른 동작 선택에 자주 쓰임. 자동화에 좋음 

- `if`는 **조건이 맞는지 아닌지**를 판단할 때
- `case`는 **값이 여러 종류 중 무엇인지**에 따라 분기할 때

### 반복문

**`for`: 정해진 목록을 반복할 때**

```
for 변수 in 값1 값2 값3
do
    실행할 작업
done
```

```
for name in alice bob charlie
do
    echo "$name"
done
```

이건 `alice`, `bob`, `charlie`를 차례로 변수에 넣어가며 반복한다

- 실무에서는 단순 숫자 반복보다 **파일 목록, 사용자 목록, 패키지 목록**처럼 `정해진 집합`을 반복 작업 할때 쓰임

**`while`**

```
while 조건
do
    실행할 작업
done
```

- 어떤 조건이 계속 유지 되는 동안 반복하는 경우
- 입력을 한 줄씩 읽는 반복문인 경우 → 실무에서 많이 쓰임

for 은 정해진 횟수, 목록 만큼 반복하는 거고 while은 조건이 참인 동안 계속 반복하는 것 

**`while read`**

```
명령어 | while read 변수
do
    처리
done
```

또는 파일 입력과 함께 이렇게도 쓴다.

```
while read 변수
do
    처리
done < 파일명
```

리눅스 명령은 대부분 결과를 **줄 단위 텍스트**로 내보내기 때문에 중요하기 때문에 셸 스크립트는 그 결과를 한 줄씩 읽어서 후속 작업을 하기 매우 좋다.

예시 

```
ipa stageuser-find --all --raw | grep " uid:" | cut -d ":" -f 2 | while read uid; do ipa stageuser-activate ${uid}; done
```

이 한 줄을 단계별로 해석하면:

1. `ipa stageuser-find --all --raw`
→ stage user 목록을 길게 출력한다
2. `grep " uid:"`
→ 그중 `uid:`가 들어간 줄만 남긴다
3. `cut -d ":" -f 2`
→ `:` 기준으로 잘라서 두 번째 필드, 즉 uid 값만 뽑는다
4. `while read uid; do ...; done`
→ 뽑아낸 uid를 한 줄씩 읽으면서 `ipa stageuser-activate ${uid}`를 반복 실행한다.

셸에서는 숫자 1부터 10까지 반복하는 것보다  **파일 내용이나 명령 결과를 한 줄씩 읽어서 처리하는 것**이 훨씬 중요하다. 한줄씩 읽는게 왜 좋은걸까 ?

for 사용했을때 

```python
for i in $(cat file.txt)
do
  echo "$i"
done
```

```python
# 결과 
apple
pie
banana
split
cherry
cake
```

while read로 사용했을때 

```python
while read line
do
  echo "$line"
done < file.txt
```

```python
# 결과 
apple pie
banana split
cherry cake
```

| 구분 | for | while read |
| --- | --- | --- |
| 기준 | 공백 | 줄 |
| 데이터 유지 | 깨질 수 있음 | 안전 |
| 실무 | 거의 안씀 | 많이 씀 |

# 5. 간단한 관리 작업 자동화 실습

### 운영 자동화를 위한 도구

1. **설치 자동화 = Kickstart**
RHEL 설치 과정 자체를 자동화하고, `%pre`·`%post` 스크립트를 통해 설치 전후 작업까지 자동으로 수행한다. Kickstart는 패키지 설치뿐 아니라 리포지토리, 네트워크, 설치 후 초기 설정까지 포함하는 초기 구축 자동화 도구다.
2. **서비스 실행 흐름 제어 = systemd**`ExecStartPre=`, `ExecStart=`, `ExecStopPost=` 같은 지점에 셸 명령이나 스크립트를 연결해 서비스 시작 전 점검, 실행 중 제어, 종료 후 정리까지 자동화한다.
3. **이벤트 기반 자동화 = 네트워크/인증 이벤트 처리**
특정 서비스 상태뿐 아니라 인증 성공·실패 같은 이벤트를 받아 조건에 따라 트래픽을 차단하거나 허용하는 식의 반응형 자동화에도 셸 스크립트를 사용할 수 있다.
4. **컨테이너 운영 자동화 = Podman + systemd**
컨테이너를 systemd unit으로 연결해 자동 시작, 중지, 재시작 같은 운영 흐름을 서비스처럼 관리할 수 있다.
5. **여러 서버 일관 관리 = system roles / Ansible**
journald, logging, storage 같은 운영 영역을 역할 단위로 표준화해 여러 호스트에 반복 가능하고 일관되게 적용한다.
